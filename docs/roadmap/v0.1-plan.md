# Milestone v0.1 — Parser, AST, Formatter (Detailed Plan)

Goal: Deliver a deterministic parser, concrete AST, and canonical formatter that normalizes source code and enables `aic check` (format-or-fail) for the MVP surface described in docs/syntax.md.

## Scope & Outcomes
- Parse the MVP language surface: modules, imports, structs, enums, type aliases, consts, lets, functions (incl. async, generics, where, effects), traits, impls, control flow (`if/else`, `loop`, `for`), `match`, basic expressions (calls, member access, literals, `await`, unary/binary ops), patterns, and blocks.
- Produce a stable, lossless AST with source spans and doc comments.
- Implement a canonical, idempotent formatter. `aic fmt` rewrites; `aic check` fails on differences.
- Emit machine-readable diagnostics (JSON) with stable codes and 1-step fix-its for common parse/format errors.
- Establish token-economy baselines and formatting rules that minimize tokens without ambiguity.

## Deliverables
- Lexer, Parser, AST crates/modules with unit tests and golden tests.
- Formatter with style rules implemented and idempotency tests.
- `aic fmt` and `aic check` subcommands wired to frontend.
- Diagnostic schema and initial error codes (E0001–E0015).
- Golden examples: all snippets from docs/vision.md and docs/syntax.md parse and format cleanly.
- Metrics script: collect token/char counts before/after formatting on sample corpus.

## Non-Goals (v0.1)
- No type checking, trait resolution, effects checking, or codegen.
- No macro system, operator overloading, or semicolon elision.

## Language Surface (MVP Parse Set)
- Files: UTF-8, newline `\n`; `//`, `/* */`, `///` comments; comments attach to nearest following item.
- Decls: `module path.name;`, `use path::item [as alias];`, `pub` visibility; `struct`, `enum`, `type`, `const`, `let`.
- Fns: `pub? async? fn name<T...>(params) -> Ret where ... effects(...)? { ... }`.
- Traits/impls: `trait` with method sigs; `impl Trait for Type { ... }`.
- Control: `if/else`, `loop`, `for x in iter {}`, `match expr { arms }`.
- Expr: literals, tuples, arrays, maps, calls, field/index access, `await`, unary/binary ops, blocks, `return`, `?`.
- Patterns: identifiers, `_`, destructuring for tuples/structs/enums.
- Semicolons required after statements; K&R braces; 4-space indents; trailing commas allowed in multiline lists.

## Grammar (Sketch, EBNF-style, excerpt)
- module_decl = "module", path, ";";
- use_decl = "use", path, ["as", ident], ";";
- item = module_decl | use_decl | struct_decl | enum_decl | type_alias | const_decl | fn_item | trait_decl | impl_block | doc_comment;
- fn_item = vis?, async?, "fn", ident, generics?, params, ret_type?, where_clause?, effects_clause?, block;
- effects_clause = "effects", "(", ident_list, ")";
- block = "{", stmt*, "}";
- stmt = let_stmt | expr_stmt | item | "return", expr?, ";";
- match_expr = "match", expr, "{", match_arm+, "}"; match_arm = pattern_list, "=>", expr_or_block, ",";
- pattern = "_" | ident | struct_pat | enum_pat | tuple_pat | lit;

Full grammar to be maintained as a source file (grammar/README.md) and mirrored in docs/syntax.md.

## AST Design
- Lossless for formatting: retain tokens necessary for canonical layout decisions; preserve doc comments (`///`) and item-level `//` when leading.
- Nodes carry: kind, children, stable NodeId, source span (file, start,end), doc string (optional), visibility, attributes (reserved).
- Keep expressions and patterns distinct; uniform literal nodes; explicit `Effects` list on functions.
- No semantic fields (types/resolution) yet; names are plain idents and paths.

## Lexer & Tokens
- Recognize keywords from docs/syntax.md; identifiers; numeric literals (int, float, hex); string/bytes; punctuation; operators.
- Handle raw strings `r"..."` and escape sequences with strict validation.
- Tokenize `_` as an identifier but allow pattern/expr hole handling in parser with context.
- Track trivia (whitespace/comments) in a side channel for doc comment association.

## Error Recovery Strategy
- Panic-mode with synchronization tokens: `;`, `}`, `]`, `)`, `,`.
- Target single-root-cause diagnostics: stop cascades after first confident error in a region.
- Common fix-its: insert missing `;`, insert missing `,`, add `}`/`)`, replace unexpected token, suggest `use` for FQNs (formatting stage only).

## Formatter Design
- Idempotent: format(parse(format(src))) == format(src).
- Statement layout: one per line; required `;`.
- Braces: same-line opens; `} else {` compact form.
- Imports: group `use` after `module`; sort by path; one per line; preserve `as alias`.
- Spaces: operators have single spaces around, none before `;`, none before `,`.
- Indentation: 4 spaces; continued lines align to 1 indent.
- Trailing commas: enabled for multiline collections and enum/struct lists.
- Blank lines: one between top-level items; none at file end; keep doc comments attached.
- Strings: do not normalize escapes (avoid semantic changes); preserve raw vs normal.

## Diagnostics (Initial Codes)
- E0001: Unexpected token
- E0002: Expected token missing (detail)
- E0003: Unclosed delimiter (which)
- E0004: Invalid numeric literal
- E0005: Invalid string/escape
- E0006: Trailing tokens after item
- E0007: Duplicate module declaration
- E0008: Misplaced `effects(...)`
- E0009: Invalid match arm (missing comma)
- E0010: Invalid pattern
- E0011: Reserved keyword used as ident
- E0012: File encoding error
- E0013: Doc comment placement error
- E0014: Formatter disagreement
- E0015: Idempotency failure

Each diagnostic includes span, message, and optional fix-it with replacement text.

## CLI & JSON Output
- `aic fmt [--check] [--json] [paths...]`
  - Writes formatted code unless `--check` is used; exits 0 if clean, 2 if disagreements.
- `aic check [--json] [paths...]`
  - Parses + formats in-memory; exits non-zero on parse errors or format disagreements; prints diagnostics.
- JSON schema versioned; include file, line/col, code, message, notes, fix-its.

## Tests & Quality Gates
- Golden files: parse and format all examples from docs; store formatted output as goldens.
- Idempotency property tests for formatter.
- Fuzzing: token-level mutator to feed parser; assert no panics; errors are reported.
- Import ordering tests; block/brace spacing tests.
- Token budget report: script prints token/char counts for examples pre/post formatting.

## Work Breakdown (2–3 weeks)
- Week 1
  - Lexer with trivia + unit tests.
  - AST scaffolding and node kinds.
  - Minimal parser for module/use/items; simple error recovery.
  - Formatter skeleton: file/module/use ordering; basic spacing.
- Week 2
  - Parse structs/enums/type aliases/const/let; functions (sig + block); control flow; expressions; patterns.
  - Expand error recovery; add diagnostics and fix-its.
  - Formatter rules for declarations, blocks, expressions; trailing commas; blank lines; idempotency tests.
- Week 3 (buffer)
  - `aic fmt` and `aic check` wiring; JSON diagnostics.
  - Golden tests for all docs; fuzzing pass; token economy baseline.
  - Polish: stability of error codes; finalize style guide; documentation.

## Risks & Mitigations
- Ambiguous grammar → keep precedence small; add explicit separators; rely on required semicolons.
- Comment loss vs formatting → preserve doc comments; defer arbitrary inline comment retention to v0.2 if needed.
- Idempotency issues → property tests and incremental formatter passes.

## Acceptance Criteria
- All examples in docs/syntax.md and docs/vision.md parse and format; `aic check` passes on them.
- `aic fmt` is idempotent on corpus; disagreements reported with E0014.
- JSON diagnostics validated against schema; codes are documented.
- Token/char metric report produced; baseline recorded in repo.
